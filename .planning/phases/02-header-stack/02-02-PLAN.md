## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local working tree → GitHub remote | `git push` exposes the commit to the public profile repo; observable by anyone visiting the profile |
| GitHub renderer → readme-typing-svg.demolab.com | GitHub fetches the SVG server-side or proxies via camo; result is what the human verifies |
| GitHub renderer → img.shields.io | Same as above for badges |
| Human visual judgement → checkpoint outcome | The human is the verifier; mis-judgement leads to false approval |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Tampering | git push to wrong remote | mitigate | Action step 2 explicitly runs `git remote -v` to confirm the remote is the profile repo before pushing. |
| T-02-09 | Repudiation | Human approval is unrecorded | mitigate | Acceptance criterion requires 02-02-SUMMARY.md to record the literal string `VISUAL-02 verified` plus date/time and themes tested — creates an auditable trail. |
| T-02-10 | Information Disclosure | Pushing to a public profile repo | accept | The README is intentionally public; that is the entire point of the phase. No secrets are present (verified by Plan 01's grep gates). |
| T-02-11 | Spoofing | Profile URL substitution by a malicious actor | mitigate | The verification URL `https://github.com/joshiaditya0511` is hard-coded in the action; the human visits this URL directly, not a link from elsewhere. |
